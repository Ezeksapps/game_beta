Renderer cannot render a frame if Scene is empty or unassigned. renderFrame() should return immediately if such a thing is attempted
a Scene is loaded with setScene, a scene can be switched out for another with setScene()
after creation, a Scene will always exist until ~Renderer() is called, because Renderer only exists in order to render a Scene

Upon setScene() being called, m_pScene should be assigned to a new unique_ptr assigned to the newly-constructed Scene
setScene will not clear the sprite texture array, but it will reset m_numSprites to zero, so anything loaded in there from the
previously-loaded Scene is treated as if it didn't exist and is overwritten as needed. This is done with registerSprite(), which
will deal with loading the new sprite object's texture data into memory at index m_numSprites, and increment the counter.

For situations where the sprite sheet being used for an entity is being changed, swapSprite()
takes the index of the old sprite you wish to replace and the new Sprite object and handles the swap

(Though if the new sprite is smaller, then the residual space in memory used up by the old sprite won't be overwritten. This could be an issue
if there is a situation where a smaller sprite sheet is loaded first, followed by a larger one, as this would overwrite the pages corresponding
to a neightbouring sprite. Also, the size of the texture array is not dynamic. Initially it was set to m_maxInstances, but that is no longer possible
with the implementation of sprite sheets as the number of entities no longer corresponds to the number of single sprite pages.
To counteract both these issues, I must first determine the maximum accepted size grid that a sprite sheet may use. Then, when loading
any sprite, regardless of size, into memory, assign the matching number of pages to the sprite, even if not all are used. The size of the array
is also now set to m_maxInstances * maxSizeWidth * maxSizeHeight. This way, Sprites never overwrite each other and there is enough space
to store all sprite sheets.
For now, the maximum number of pages (frames) a sprite sheet may hold is 10 * 10. This might be adjusted later)

The Sprite struct is:

struct Sprite {
    std::string filepath;
    vec3 pos;                       // Position (before accounting for world-view-model matrix)
    int index;                      // start index in tex array
    int pages;                      // number of pages taken up by this spritesheet
    std::vector<int> pageDurations; // duration of each page/frame for one cycle of the animation
    int pagesPerAnim;               // pages taken up by each animation
    // (endPage = index + pages)
};

a Sprite refers to a single sprite sheet. Index is not the 'number' of this entity, it is the start index in the whole
texture array of the first page (frame) in the sprite sheet this Sprite object represents.
each page has a duration in milliseconds, which is given by the pageDurations vector.
Each sprite sheet contains the frames to animate a specific action for the Entity. This is repeated for all
possible directions the Entity may be facing, which are north, north-east, east, south-east, south, south-west, west and north-west.
(This means every sprite sheet has a maximum of 8 rows)
Since every row is just a different version of the animation for a particular direction, the number of frames per row is also uniform.
pageDurations is therefore the same size as the number of frames in a row.
The pagesPerAnim field may not be necessary, since it can be calculated by the PNG width divided by 192 (all sprites are 192 x 192 px)

Entity holds an enum AnimEvent, which represents the events an entity can be in. The JSON for an Entity holds the page durations and the AnimEvents for an entity
as well as the filepaths to the PNG sprite sheets for these events. This JSON is loaded in Entity's constructor and fills the sprite map.
The sprite map has the AnimEvent as the key and the Sprite object as the value. The private filed m_pActiveSprite defines the active animation for the Entity.

This is all implemented, which leaves these things to deal with:
Entity needs to store the direction it is facing. The correct row of the Sprite can then be used.
The rows corresponding to every direction must be defined as a constant.
The renderer must count the frame (col) in the row that is currently being shown and only keep it rendered for the specified time period
(This means frame rate now needs to be controlled and taken into account), then loop back to the beginning
